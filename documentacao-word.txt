5. SISTEMA DE CACHE
========================================

O CACHE É UMA "MEMÓRIA TEMPORÁRIA" DO SITE:

POR QUE EXISTE:
- Buscar dados do WordPress demora alguns segundos
- Com cache, o site fica muito mais rápido
- Evita sobrecarregar o WordPress com muitas consultas
- Reduz tempo de carregamento de 3-5s para 0.5s

COMO FUNCIONA:
1. Primeira visita: Site busca dados no WordPress (demora um pouco)
2. Site salva esses dados na pasta "cache"
3. Próximas visitas: Site usa dados salvos (muito rápido)
4. Após 30 minutos: Cache expira, site busca dados novos

ONDE FICA O CACHE:
📁 cache/
├── 22f0fa8d672dcb7afe603fa38897d811.cache → Dados do banner
├── a1b2c3d4e5f6789012345678901234ab.cache → Fotos da cultura
├── b2c3d4e5f6789012345678901234abcd.cache → Números da atuação
└── [outros arquivos .cache]

COMO LIMPAR O CACHE (quando conteúdo não atualiza):

MÉTODO 1 - Painel de Controle:
1. Acesse o painel da sua hospedagem
2. Vá no gerenciador de arquivos
3. Navegue até "brasilcenter/cache"
4. Selecione todos os arquivos .cache
5. Delete os arquivos
6. Pronto! Próxima visita buscará dados novos

MÉTODO 2 - Via FTP:
1. Abra seu programa de FTP
2. Conecte no servidor
3. Vá na pasta "cache"
4. Delete todos os arquivos .cache

IMPORTANTE:
- Delete APENAS arquivos .cache
- NÃO delete a pasta cache/ em si
- Após limpar, primeira visita será mais lenta
- Cache se recria automaticamente

7. API E ENDPOINTS
========================================

CONEXÃO COM WORDPRESS:

ENDPOINT PRINCIPAL:
https://www.brasilcenter.com.br/wordpress/graphql

ARQUIVO RESPONSÁVEL:
📄 includes/graphql-client.php

COMO FUNCIONA A COMUNICAÇÃO:
1. Site faz "pergunta" para WordPress via GraphQL
2. WordPress responde com dados em formato JSON
3. Site processa os dados e monta a página
4. Dados ficam salvos no cache por 30 minutos

FUNÇÕES DISPONÍVEIS:
- get_home_banner() → Busca dados do banner principal
- get_home_cultura() → Busca fotos da seção cultura
- get_home_atuacao() → Busca números da empresa
- get_sobre_banner() → Busca banner da página sobre
- get_sobre_conquistas() → Busca prêmios e conquistas
- get_jeito_bcc_banner() → Busca banner da página jeito BCC

CONFIGURAÇÕES DA API:
- Timeout: 15 segundos (se não responder, desiste)
- Cache: 30 minutos por consulta
- Retry: 3 tentativas em caso de falha
- Fallback: Exibe conteúdo padrão se API falhar

EXEMPLO DE CONSULTA:
Site pergunta: "Me dê os dados do banner da home"
WordPress responde: {
  "title": "Bem-vindo à BrasilCenter",
  "text": "Conectando talentos...",
  "image": "banner-home.jpg"
}

MONITORAMENTO:
- Logs de erro salvos automaticamente
- Alertas em caso de falha na API
- Métricas de performance disponíveis

9. HANDTALK (ACESSIBILIDADE)
========================================

O HandTalk é um plugin de acessibilidade que traduz o conteúdo do site para Libras (Língua Brasileira de Sinais).

ONDE ESTÁ INSTALADO:
📄 includes/head-config.php (script de inicialização)

COMO ESTÁ CONFIGURADO:
- Plugin carrega automaticamente em todas as páginas
- Ícone azul aparece no canto inferior direito
- Tradução em tempo real do conteúdo

TOKEN ATUAL:
Localização: includes/head-config.php
Procure por: ht_token: "XXXXXXXXXX"

COMO ALTERAR O TOKEN:
1. Abra: includes/head-config.php
2. Procure pela linha: ht_token: "XXXXXXXXXX"
3. Substitua pelo novo token fornecido pela HandTalk
4. Salve o arquivo
5. Limpe o cache do site

FUNCIONALIDADES ATIVAS:
- Tradução automática de textos
- Avatar em Libras
- Controles de velocidade
- Modo de alto contraste
- Navegação por teclado

CONFIGURAÇÕES ATUAIS:
- Posição: Canto inferior direito
- Cor do ícone: Azul padrão HandTalk
- Tamanho: Responsivo
- Idioma: Português brasileiro

SUPORTE TÉCNICO HANDTALK:
- Site: https://www.handtalk.me/
- Email: suporte@handtalk.me
- Documentação: https://developers.handtalk.me/

RENOVAÇÃO DO TOKEN:
- Tokens têm validade anual
- HandTalk avisa 30 dias antes do vencimento
- Renovação feita diretamente no painel HandTalk
- Após renovar, atualizar no arquivo head-config.php

IMPORTANTE:
- Token é único para cada domínio
- Não compartilhar o token publicamente
- Testar funcionamento após alterações
- Manter token sempre atualizado

10. WEB.CONFIG E SUA IMPORTÂNCIA
========================================

O arquivo web.config é fundamental para o funcionamento correto do site em servidores Windows/IIS.

LOCALIZAÇÃO:
📄 web.config (na raiz do site)

FUNÇÃO PRINCIPAL:
- Configura regras de redirecionamento (URL Rewrite)
- Define configurações de segurança
- Controla cache de arquivos estáticos
- Gerencia compressão de conteúdo

PRINCIPAIS CONFIGURAÇÕES:

URL REWRITE:
- Redireciona URLs amigáveis
- Remove extensão .php das URLs
- Força HTTPS (se configurado)
- Redireciona www para não-www (ou vice-versa)

CACHE DE ARQUIVOS:
- CSS: Cache por 7 dias
- JS: Cache por 7 dias
- Imagens: Cache por 30 dias
- Fontes: Cache por 365 dias

COMPRESSÃO:
- Compressão gzip habilitada
- Reduz tamanho dos arquivos em 60-80%
- Melhora velocidade de carregamento

SEGURANÇA:
- Headers de segurança configurados
- Proteção contra ataques comuns
- Controle de acesso a arquivos sensíveis

QUANDO PODE DAR ERRO:

ERRO 500 (Internal Server Error):
- web.config com sintaxe incorreta
- Regras conflitantes
- Módulos não instalados no servidor

ERRO 404 (Página não encontrada):
- Regras de rewrite incorretas
- Caminhos de arquivo errados
- URL Rewrite não funcionando

PÁGINAS NÃO CARREGAM CSS/JS:
- Regras de cache muito restritivas
- Tipos MIME não configurados
- Compressão causando conflitos

COMO RESOLVER PROBLEMAS:

1. FAZER BACKUP:
   - Sempre salve uma cópia do web.config funcionando
   - Antes de qualquer alteração

2. TESTAR SINTAXE:
   - Use validadores online de web.config
   - Teste uma regra por vez

3. VERIFICAR LOGS:
   - Logs do IIS mostram erros detalhados
   - Event Viewer do Windows também ajuda

4. ROLLBACK:
   - Se der erro, volte para versão anterior
   - Teste novamente após correção

ESTRUTURA BÁSICA:
```
<?xml version="1.0" encoding="UTF-8"?>
<configuration>
  <system.webServer>
    <rewrite>
      <rules>
        <!-- Regras de redirecionamento -->
      </rules>
    </rewrite>
    <staticContent>
      <!-- Configurações de cache -->
    </staticContent>
  </system.webServer>
</configuration>
```

IMPORTÂNCIA:
- Sem web.config, site pode não funcionar
- URLs podem quebrar
- Performance pode ser prejudicada
- Segurança fica comprometida

MANUTENÇÃO:
- Revisar configurações semestralmente
- Atualizar regras conforme necessário
- Manter backup sempre atualizado
- Documentar mudanças realizadas
